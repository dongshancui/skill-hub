# Harness Engineering：各阶段的注意力优先级

> 解决的问题：各阶段 agent 的注意力应该优先投在哪里？  
> 不是讲"产出什么文档"，而是讲"什么思路决策是最关键的"。  
> 与 harness-engineering-design-vs-dev.md（分工）、harness-dev-lifecycle-guide.md（实践清单）互补。

---

## 核心判断

每个阶段最关键的东西，不是做了多少，而是**在最危险的地方有没有思考到位**。

| 阶段 | 最危险的失误 | 对应的第一优先级 |
|---|---|---|
| 设计 | 对现有代码库的错误假设 | 约束发现（什么不能动、什么已存在） |
| 开发（任务开始） | 用错误的模式实现正确的逻辑 | 微观对齐（读周边代码，找正确工具） |
| 开发（任务执行） | 改了太多、改了不该改的 | 原子执行 + 自测修复小循环 |

---

## 设计阶段：约束发现优先，方案生成其次

### 为什么不是"业务思路优先"

业务需求通常在 ticket 或需求文档里写得比较清楚，agent 理解错的概率相对低。  
真正高频的失误是：**设计了一个跟现有系统冲突的方案，或者重新实现了已经有的东西。**

这类失误的代价极高——开发阶段才发现，等于设计做了两遍。

### 正确的设计思路顺序

```
Step 1: 约束边界（必须先做）
  → 现有代码里什么不能动？（共享表、公共接口、被多处引用的服务）
  → 哪些架构决策已经固化了？（分层方式、事务边界、错误处理模式）
  → 有没有正在进行中的改动会影响这次设计？

Step 2: 业务意图（What，不是 How）
  → 做完了，"正确"长什么样？（可测试的验收标准）
  → 边界条件和异常场景是什么？
  → 什么情况明确不在本次范围内？

Step 3: 复用机会（在约束和意图都清楚之后才谈）
  → 现有代码里有没有可以直接接上的接口/工具/能力？
  → 复用还是新建，决策标准是什么？（改现有的风险 vs. 造新的重复）

Step 4: 方案约束（不是实现路径，是解空间边界）
  → 数据结构约束（必须新建的表/字段，必须复用的现有结构）
  → 核心业务逻辑约束（编码业务规则的 SQL/伪代码）
  → 验收标准（Evaluator Agent 用于验证的具体条件）
```

### 关键原则：设计文档管的是约束空间，不是实现路径

业务思路和特性复用都是设计输入，不是设计输出。  
设计输出是：**什么必须为真，什么不能被违反，什么可以复用。**

"现有特性复用"和"关键业务思路"不是二选一，而是前后两步：
- 没有业务意图，就不知道复用什么（可能复用了错误的东西）
- 没有现状分析，就不知道哪里有东西可复用

## 开发阶段：三层时机，各有重心

开发阶段的最大问题，是把三个不同时机的活动混在一起做：

```
❌ 错误做法：边探索边实现，"找到什么就改什么"
✓  正确做法：明确区分三个时机，每个时机只做一件事
```

### 时机一：每个原子任务开始前——微观对齐（最容易被跳过）

这是开发阶段缺失最多的一步。  
它不是"探索项目"（太宽，会造成上下文污染），也不是直接写代码（太急，会写出格格不入的东西）。

**微观对齐的操作**：在改某个文件之前，先读它的"邻居"：

```
1. 读要修改的文件本身（掌握现有结构和模式）
2. 读同层的 2-3 个相似实现（同一 Service 层里其他方法是怎么写的）
3. 找这个模块里用到的工具类/工厂/帮助方法（避免重新实现已有能力）
4. 看最近 3-5 个 commit 改了什么（有没有正在进行中的重构）
```

**微观对齐解决的问题**：
- 代码风格与周边一致（命名、异常处理、日志格式）
- 自动发现可复用的工具方法（不用在设计阶段预测所有细节）
- 避免与"正在进行中"的改动产生冲突

**微观对齐不做的事**：
- 不读与本任务无关的模块
- 不在这步做架构层面的判断（那是设计阶段的事）
- 不开始写代码（还没到时候）

### 时机二：原子任务执行中——实现 + 自测修复小循环

这一步现有文档（harness-dev-lifecycle-guide.md）覆盖得比较好。  
核心原则：**只做 solution-design.md 里定义的事，发现超出范围的问题就暂停。**

自测修复的节奏：
```
写代码 → 编译（发现语法/类型错误）→ 跑相关单测 → 修复失败 → 再跑 → 通过后提交
```

每次提交前，agent 输出变更摘要（改了什么、没改什么、发现了什么问题）。

### 时机三：原子任务完成后——边界检查

防止"完成了这个任务，但影响了不该影响的东西"：

```
1. 运行整个模块的测试（不只是刚写的测试）
2. 输出本次 diff 的文件列表，对照任务分解里的预期文件
3. 如果多改了文件：停下来，说明原因，等人工确认
```

## "项目探索"应该在哪个时机做

| 探索类型 | 正确时机 | 深度 |
|---|---|---|
| 服务/模块级别的影响面分析 | 设计阶段（需求分析步骤） | 宽，浅 |
| 相关模块的代码现状分析 | 设计阶段（代码 baseline 步骤） | 中，中 |
| 要修改的具体文件 + 周边代码 | 每个原子任务开始前（微观对齐） | 窄，深 |
| 发现的无关问题/Bug | 随时，但只记录，不处理 | 不展开 |

**如果开发阶段出现了大量"探索"，通常说明设计阶段的代码 baseline 做得不够深。**  
解决方式不是在开发时补做，而是在设计阶段 checklist 里强制要求更深的现状分析。

## 理解和遵从规范：应该是前提，不是阶段活动

"对项目现有工具类、开发规范的理解"本质上是一个**前提条件**，不是某个阶段的主要活动。

正确的处理方式：把它变成 AGENTS.md 里的持久化约束，而不是每次任务都重新"学习"：

```markdown
# AGENTS.md 中规范的正确写法

## 工具类使用（看代码，不看描述）
禁止自行实现日期格式化，使用项目已有的 DateUtils.format()：
  → 路径：common/src/main/java/com/xx/util/DateUtils.java

禁止自行实现分页参数解析，使用：
  → 路径：common/src/main/java/com/xx/web/PageableUtils.java

## 错误处理规范（看代码例子）
参考：OrderService.java:87-95 的异常抛出方式
```

这样，微观对齐时 agent 读周边代码，加上 AGENTS.md 里的约束，规范遵从是自然结果，不是额外的"阶段"。

## 一张图：完整的注意力分配

```
需求 Brief 进来
    │
    ▼
【设计阶段】
  第一步：约束边界（现有系统里什么不能动）
  第二步：业务意图（AC、边界条件）
  第三步：复用机会（在约束和意图都清楚后）
  第四步：方案约束文档（供 agent 开发用）
    │
    ▼
【每个原子任务开始时】（微观对齐，容易被跳过）
  读要改的文件 + 周边 2-3 个同层文件
  找可用的工具方法
  看最近相关 commit
    │
    ▼
【原子任务执行中】
  写代码 → 编译 → 跑测试 → 修复 → 提交
  发现超出范围的问题 → 记录、暂停、等确认
    │
    ▼
【原子任务完成后】
  跑整个模块测试
  输出 diff 文件列表对照预期
    │
    ▼
  全部任务完成 → 人工验收 → 对照 AC
```

## 小结

| 问题 | 答案 |
|---|---|
| 设计阶段最重要的是现有特性复用还是业务思路？ | 都要，但有顺序：先约束边界，再业务意图，最后复用机会 |
| 开发阶段最重要的是US拆解+自测循环还是项目探索？ | 项目探索应该在设计阶段做完；开发阶段加一步"微观对齐"（任务级别的局部代码读取）再进入实现循环 |
| 遵从开发规范应该在哪个阶段做？ | 不是阶段活动，是前提条件——写进 AGENTS.md，通过微观对齐时读周边代码自然生效 |
| 如果开发阶段探索很多，说明什么？ | 设计阶段的代码 baseline 分析做得不够深，下次在设计 checklist 里加强 |
